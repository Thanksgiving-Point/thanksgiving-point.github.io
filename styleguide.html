Engineering Style Guide

Internal coding standards for Thanksgiving Point exhibits firmware. This guide keeps our work readable, maintainable, and consistent as our systems grow.

1. Purpose

This lightweight style guide establishes consistent formatting, naming, and documentation practices across all firmware projects. Made with an emphasis on maintainability rather than strict guidelines.

2. Repository Structure

  • src/: All firmware source files (.cpp, .h)
  • hardware/: Wiring diagrams, schematics, power notes
  • docs/: maintenance notes, troubleshooting guides
  • No per-class subfolders (for Arduino IDE compatibility).

3. File Header Template

3.1 Include at the top of every main .ino file:

/**
 * File: <filename>
 * Project: <project title>
 * Author: <name>
 * Date: <mm-dd-yy>
 * Description: <short description of this file>
 * Notes: <dependencies, pin usage, modules, quirks, etc.>
 * (c) Thanksgiving Point Exhibits Electronics Team — <year>
 */

3.2 Include at the top of every .h file: 

/**
 * <class_name>
 * 
 * <descriptive one liner and list if necessary>
 * Trigger-based audio playback for MKR Zero.
 * - Exposes play()
 * - Manages busy state and pulse timing
 * - Non-blocking update() method
 */

4. Naming Conventions

4.1 Variables

  • Local variables: camelCase
  • Member variables: camelCase_ (trailing underscore)
  • Global constants: SCREAMING_SNAKE_CASE

4.2 Types

  • Classes: PascalCase
  • Structs: also PascalCase
  • Enums Type Name: PascalCase
  • Enums Member Name: SCREAMING_SNAKE_CASE

5. Functions and Methods

5.1 Documentation

Each function includes a one-line summary + parameters + return value.

/**
 * @brief description of what the function does.
 * more detailed explanation if needed
 *
 * @param <value> description of parameter
 * @return <value> description of return value
 */
bool camelCase(int param);

5.2 Formatting

  • 2-space indentation (clang-format)
  • Automatic formatting enforced before commits
  • Line length: determined by clang-format (usually 100–120 chars)

6. Inline Comments

Inline comments are encouraged when clarity is needed. Prefered to:

  • Explain why something is done, not just what it does
  • Avoid stating the obvious (“increment i”)
  • Use them to document hardware/software quirks, timing constraints, electrical behavior, etc.

Inline comment density is not formally required — use judgment and ensure clarity for future maintainers.

7. File Organization

  • Header files contain declarations + short inline functions
  • Source files contain implementations/definitions
  • Try to keep hardware pin definitions and constants in Config.h

8. Versioning and Tags

Use descriptive tags for important project milestones, e.g.:

v0.9-mkrzero-i2s-final
v1.0-sound-module-refactor

9. Hardware Documentation

For hardware/ folder

  • Document wiring, connector types, calculated power requirements/ratings of components
  • Define startup quirks or power/reset procedures
  • Calibration steps
  • Include schematics pdf if possible

10. README.md Template

  • Overview: 5-10 sentence summary
  • Hardware List: microcontrollers, sensors, power supplies, etc.
  • Wiring Diagram
  • Software Architecture: key classes only and data flow description if too complex
  • Troubleshooting & known issues and warnings
