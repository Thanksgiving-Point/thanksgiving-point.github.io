Engineering Style Guide

Internal coding standards for museum firmware projects. This guide keeps our work readable, maintainable, and consistent as our systems grow.

1. Purpose

This lightweight style guide establishes consistent formatting, naming, and documentation practices across all museum microcontroller firmware projects (Arduino, ESP32, RP2040, Teensy, etc.). It is designed for a small engineering team with a strong emphasis on maintainability.

2. Repository Structure

  • All firmware lives in a single src/ directory.
  • Custom classes use a paired .h and .cpp file in that folder.
  • No per-class subfolders (Arduino IDE compatibility).

3. File Header Template

Include at the top of every .ino, .h, and .cpp file:

/**
 * @file
 * @brief Short 1–2 sentence summary of the file's purpose.
 *
 * Detailed description if needed. Mention hardware interfaces,
 * expected behavior, and any assumptions.
 *
 * (c) Museum Electronics Team — <year>
 */

4. Naming Conventions

4.1 Variables

  • Local variables: camelCase
  • Member variables: camelCase_ (trailing underscore)
  • Global constants: kConstantName
  • Macros (avoid): ALL_CAPS_LIKE_THIS

4.2 Types

  • Classes: PascalCase
  • Structs: also PascalCase
  • Enums + enum values: kEnumValueName

5. Functions and Methods

5.1 Documentation

Each function includes a one-line summary + parameters + return value.

/**
 * @brief Returns true if the battery voltage is above threshold.
 *
 * @param reading Raw ADC reading (0–4095).
 * @return true if voltage is acceptable.
 */
bool isBatteryHealthy(int reading);

5.2 Formatting

  • 2-space indentation (clang-format)
  • Automatic formatting enforced before commits
  • Line length: determined by clang-format (usually 100–120 chars)

6. Inline Comments

Inline comments are encouraged when clarity is needed. We prefer:

  • Explain why something is done, not just what it does
  • Avoid stating the obvious (“increment i”)
  • Use them to document hardware quirks, timing constraints, electrical behavior, etc.

Inline comment density is not formally required — use judgment and ensure clarity for future maintainers.

7. File Organization

  • Header files contain declarations + short inline functions
  • Source files contain implementations
  • Keep hardware pin definitions and constants in Config.h

8. Versioning and Tags

Use descriptive tags for important project milestones, e.g.:

v0.9-mkrzero-i2s-final
v1.0-sound-module-refactor

9. Hardware Notes

  • Document wiring, connector types, fuse ratings
  • Define startup quirks or power ordering requirements
  • Keep schematics in hardware/ folder where possible
